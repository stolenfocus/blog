## 오늘의 결정 사항

엄마/아빠 각자 명의의 한투 연금저축 계좌 두 개를 운용하고 있습니다. 6종 ETF 분산(코스피200 25%, 금현물 25%, S&P500 20%, 신재생 15%, 인도 10%, 원유 5%) + 분기 임계치 리밸런싱 + 매월 적립.

오늘 같이 검토하면서 정리된 결정:

### 1. 임계치 3%p → 5%p 전환 (확정)

기존엔 3%p였습니다. 이게 옛날 백테(순수 리밸만 시뮬)에서 최적이었거든요. 그런데 실제 운용은 **DCA + 분기 리밸**이 결합된 형태인데, 그 결합 시나리오는 백테한 적이 없었습니다.

결합 백테(11년) 돌려보니:

| 임계치 | CAGR | 리밸 횟수 |
|---|---|---|
| 1%p | 10.18% | 34회 |
| 3%p (기존) | 10.06% | 10회 |
| **5%p** | **10.38%** | **6회** |
| Smart DCA only | 9.96% | 0회 |

5%p가 32bp 우위. 이유는 명확합니다 — DCA 자체가 underweight를 자연스럽게 채우는 부드러운 리밸 역할을 하기 때문에, 위에 3%p로 또 자주 트리거하면 winners를 너무 일찍 자르는 셈이 됩니다. 5%p로 충분히 달리게 두고 결정적인 순간에만 정렬하는 쪽이 +α가 나옵니다.

`scripts/trading_kr/config.py:48`의 `REBALANCE_THRESHOLD = 0.03 → 0.05` 한 줄 변경으로 적용. 다음 분기 게이트(2026-07-01)부터 발효.

### 2. 월 적립금 30만 → 50만 (방향 확정, 부모님 cash flow 확인 대기)

DCA 금액 sweep 결과 흥미로운 패턴이 나왔습니다.

| 월 적립 | 5년 후 평가/계좌 | CAGR |
|---|---|---|
| 30만 (현재) | 약 4,200만 | 9.40% |
| 50만 (세제캡) | 약 5,400만 | 9.23% |
| 100만 | 약 7,200만 | 9.09% |
| 150만 (연금캡) | 약 9,400만 | 9.04% |

**월 적립이 늘수록 CAGR(%)은 살짝 떨어지지만 절대 수익 금액은 4배 차이**가 납니다. CAGR이 떨어지는 건 손해가 아니라 "늦게 들어간 돈은 짧게 굴러서 평균이 희석"되는 수치 효과입니다. 운용 효율 자체는 같습니다.

결국 "월 얼마"는 **운용 전략 문제가 아니라 cash flow 문제**라는 게 정리됐습니다. 가용한 만큼 빠르게 deploy하는 게 답이고, 연금저축 cap이 월 150만(연 1,800만)이니 그게 합법적 최대 속도.

세제혜택이 월 50만(연 600만)까지라 그 라인이 가장 직관적인 sweet spot. 부모님 매월 50만 입금 가능 여부 확인되면 즉시 전환 예정.

### 3. 인출기 전략 잠정 — 5년 후 배당주 tilt

5년 운용 후 인출 단계를 어떻게 디자인할지 비어 있던 부분이 있었습니다. 같이 정리한 잠정 방향:

- **운용기 (지금~2031)**: 6종 성장 그대로, 매월 DCA
- **인출기 (2031~)**: 배당주 ETF (TIGER 미국배당다우존스 = SCHD형, KODEX 한국배당 등)로 점진 tilt → **분배금 자체가 인출 cash flow** 역할

배당 vs 성장은 양자택일이 아니라 단계 전환의 문제입니다. 연금저축 안에선 어차피 과세이연이라 배당과 성장이 세금 측면에서 동등합니다. 운용기엔 성장 중심으로 키우고, 인출기엔 분배금으로 원금 안 까먹는 구조를 만드는 쪽이 4% rule보다 안전합니다.

## 그리고 발견한 백테 버그 두 개

오늘 결합 백테를 돌리던 중 절대 금액 수치가 직관과 어긋나서 검증해보니 두 가지 버그가 있었습니다.

### 버그 1 — DCA 날짜 매칭

```python
# 버그 있는 코드
month_starts = prices.resample("MS").first().index
dca_dates = set(prices.index[prices.index.isin(month_starts)][1:])
```

`resample("MS").first().index`는 **캘린더 1일** 타임스탬프를 반환합니다. 예: `2015-01-01`, `2015-02-01`, ...

문제는 **캘린더 1일이 영업일이 아닐 때**입니다. 1월 1일은 휴일, 5월 1일은 노동절, 종종 토/일요일 — 이런 날짜는 `prices.index`(영업일만 있음)에 없으니 `.isin()`이 매칭 안 됨.

결과: 11년 데이터 136개월 중 **DCA가 68회만 발생** (절반!).

### 버그 2 — 분기 리밸 게이트

```python
# 버그 있는 코드
rebal_dates = set(prices.resample("QE").last().index)
```

같은 패턴의 변형. `QE`(quarter-end)는 캘린더 분기말(3/31, 6/30, 9/30, 12/31). 이 날짜들이:
- 주말일 때 누락
- 12/31은 KRX 종무식이라 매년 거래 안 함 → 누락
- 일부 9/30은 추석과 겹쳐 누락

11년 데이터 46개 분기말 중 **22개(약 48%)가 누락**됐습니다. 즉 "분기 1회 리밸 체크"가 실제로는 "분기 2회 중 1회만 체크"였던 셈.

### 수정 패턴

```python
# 수정된 코드
idx_s = pd.Series(prices.index, index=prices.index)
dca_dates = set(
    idx_s.groupby(pd.Grouper(freq="MS")).first().dropna().tolist()
)
rebal_dates = set(
    idx_s.groupby(pd.Grouper(freq="QE")).last().dropna().tolist()
)
```

`pd.Grouper`로 그룹핑한 뒤 각 그룹의 첫(또는 마지막) **실제 데이터 인덱스**를 가져옵니다. 그럼 영업일 안에서 가장 가까운 1일/말일이 잡힙니다.

### 재검증 — 결정은 안 바뀜

| 항목 | 버그 전 | 버그 후 |
|---|---|---|
| Pure rebal LONG 11년, 3%p | 12.04% | 12.16% |
| Pure rebal LONG 11년, 5%p | 12.00% | 12.10% |
| DCA 결합 LONG 11년, 5%p | 11.09% | 10.38% |
| DCA 결합 LONG 11년, 3%p | 10.79% | 10.06% |

CAGR 시프트는 ±50bp 이내. 5%p가 DCA 결합 시나리오에서 우위라는 결론은 동일합니다.

흥미로웠던 건 — **분기 게이트의 절반이 누락됐는데 CAGR 차이가 12bp밖에 안 났다**는 점. 직관적으로는 "절반 빠지면 결과가 망가져야" 할 것 같은데 그렇지 않습니다. 이유:

1. 게이트는 *체크 기회*일 뿐, 드리프트가 임계치 미달이면 어차피 리밸 안 함
2. 누락된 게이트의 드리프트는 **사라지지 않고 다음 게이트로 이월**됨
3. 분기 1회는 원래도 듬성한 게이트라 몇 개 빠진다고 결정적으로 안 바뀜
4. DCA가 자체적으로 드리프트를 부드럽게 줄이고 있음

리밸 횟수는 의미 있게 변했습니다 (3%p의 경우 ~5회 → 14회로 약 3배). 즉 버그가 "리밸 빈도"는 절반으로 만들었지만 "최종 수익률"에 미치는 영향은 작았던 거죠.

## 회고

### 잘한 것

- 절대 금액 수치 (60만/년 적립이 11년에 36M 투자라고 나온 시점)에서 직관과 어긋남을 알아채고 검증한 점. "60만 × 12 × 11 = 7,920만이어야 하는데 36M이라고?" 라는 단순 산수가 버그를 잡아냈습니다.
- 첫 버그 fix 후 한 번 더 audit해서 두 번째 버그까지 찾은 점.
- 다른 포폴들 전수 점검까지 한 점.

### 못한 것

- 처음 백테 코드 작성 시 `resample` 결과의 인덱스가 캘린더 기준이라는 걸 가정 없이 사용한 점. pandas 문서를 안 봤습니다.
- 검증 단계에서 단위 sanity check (투자 누적이 예상치와 일치하는지)를 안 한 점. 수치가 그럴듯해 보여서 그냥 넘겼습니다.

### 교훈

- **백테 결과는 항상 단위 산수로 한 번 검증**. CAGR/Sharpe 같은 비율 지표는 버그가 숨기 쉽고, 절대 금액(투자 누적, 거래 횟수)이 더 잘 드러냅니다.
- **resample().index와 원본 .index의 정렬 가정은 데이터 빈도를 모를 때 위험**. 영업일 데이터엔 캘린더 기반 resample 인덱스를 매칭하지 말 것.
