## 버그 3건

### 버그 1: config.py 종목 미수정

Day 1에서 ETF를 KODEX고배당주로 교체했지만, `config.py`에는 여전히 SOL코리아고배당(0105E0)이 들어있었습니다.

```python
# Before (잘못됨)
"0105E0": {"name": "SOL코리아고배당", "target_weight": 0.20}

# After (수정)
"279530": {"name": "KODEX고배당주", "target_weight": 0.20}
```

### 버그 2: daily_buy_plan.py 별도 포트폴리오

`daily_buy_plan.py`(레거시 스크립트)에 PORTFOLIO가 하드코딩되어 있었습니다. config.py를 import하지 않고 자체적으로 종목을 정의하고 있었기 때문에, config를 고쳐도 이 스크립트는 여전히 틀린 종목(PLUS고배당 161510)으로 동작했습니다.

**교훈: 포트폴리오 정보는 한 곳에서만 관리하고, 나머지는 import해서 사용해야 합니다.**

### 버그 3: buy-plan이 계획 생성과 동시에 state 업데이트

가장 심각한 버그였습니다. `buy-plan` 커맨드가 매수 계획을 보여주면서 **동시에 state.json을 업데이트**했습니다. 아직 매수하지 않았는데 이미 산 것처럼 기록되는 문제입니다.

결과:
- Day 1만 실제 매수했는데 state에는 Day 3까지 기록됨
- 보유 수량이 실제와 불일치
- 다음 계획 생성 시 잘못된 보유량 기준으로 계산

## 수정 사항

### 1. 종목 통일

모든 파일에서 SOL코리아고배당(0105E0), PLUS고배당(161510) → **KODEX고배당주(279530)**로 통일했습니다.

수정 파일:
- `scripts/dividend/config.py` — PORTFOLIO, INITIAL_HOLDINGS, DIVIDEND_RECORD_DATES
- `scripts/dividend/daily_buy_plan.py` — PORTFOLIO, EXISTING, defaults
- `scripts/dividend/manager.py` — defaults
- `DIVIDEND_STRATEGY.md` — 최종 포트폴리오 테이블
- `data/dividend_state.json` — Day 1 실제 매수분으로 리셋

### 2. plan/confirm/execute 분리

기존: `buy-plan` = 계획 생성 + state 업데이트 (위험)

변경:
- `buy-plan` → 계획만 생성, `dividend_pending_plan.json`에 임시 저장
- `confirm` → 수동 매수 후 state 반영
- `execute` → **KIS API로 시장가 자동 매수** + state 반영

### 3. 고가 종목 처리

삼성화재우(1주 ~37.9만원)는 일일 종목 배정(30만원)으로 매수 불가. 매일 잔돈만 쌓이는 문제가 있었습니다.

변경:
- Day 2~4: 고가 종목 제외, 나머지 4종목에 균등 분배
- Day 5(마지막날): 전체 비중 기반 조정 매수 + 추가금 안내

### 4. 중복 실행 방지

```python
if state.get("initial_buy_day", 0) >= pending["day"]:
    print(f"Day {pending['day']}는 이미 실행됨. 중복 매수 방지.")
    return
```

## 토큰 발급 이슈

KIS API 토큰 발급은 **1분에 1회** 제한이 있습니다. 가격 조회와 주문이 서로 다른 토큰 경로를 사용하면서 연속 발급 시도로 403 에러가 발생했습니다.

해결: PyKis의 `keep_token` 경로를 통일해서 토큰을 재사용하도록 수정.

## 오늘의 교훈

1. **포트폴리오 정보는 Single Source of Truth.** 여러 파일에 하드코딩하면 반드시 불일치가 생깁니다.
2. **계획(plan)과 실행(execute)은 반드시 분리.** "보여주기만 하는" 함수가 상태를 바꾸면 안 됩니다.
3. **ISA 실제 잔고와 코드 state를 주기적으로 대조.** API로 쉽게 할 수 있습니다.
4. **바이브 코딩의 함정**: AI가 빠르게 만들어주면 "되는 것 같아서" 검증 없이 넘어가기 쉽습니다. 돈이 오가는 코드는 반드시 시뮬레이션 + 실제 대조를 해야 합니다.
